## Fase 1 — SQL: `sql/002_proofly_bridge.sql`

Rodar no SQL Editor do Supabase Proofly. **Sem DROP. Sem 001_nucleo.**

### 1.1 Coluna opcional em qr_codes
```sql
ALTER TABLE public.qr_codes
  ADD COLUMN IF NOT EXISTS used_at TIMESTAMPTZ;
```

### 1.2 RPC validate_qr_token(p_token TEXT)
- SELECT `qr_codes` WHERE `token = TRIM(p_token)`
- JOIN `professionals` para `id`, `name`, `specialty`
- `invalid` se não existe
- `used` se `used_at IS NOT NULL` OU EXISTS review com `qr_token = p_token`
- `expired` se `expires_at < NOW()`
- `valid` + `professional_id`, `professional_name`, `professional_specialty`

Retorno JSON deve usar chaves que o JS novo já espera (`professional_slug` → **enviar `professional_id` como string** OU adaptar JS para `professional_id` — ver Fase 2).

### 1.3 RPC submit_qr_review(p_token, p_rating, p_comment)
- `FOR UPDATE` em `qr_codes`
- Revalidar expired/used/invalid
- INSERT em `reviews`:
  - `professional_id`, `rating`, `comment`
  - `verified = true`, `is_verified = true`
  - `qr_token = p_token`
  - `review_type = 'client_to_professional'`
  - `source = 'cliente'`
  - `user_id = NULL`
- UPDATE `qr_codes SET used_at = NOW()` WHERE id = ...
- Trigger existente `recalc_prof_talent_metrics` recalcula `avg_rating` — **não criar trigger novo**
- `GRANT EXECUTE` TO `anon`, `authenticated`

### 1.4 RPC create_qr_session(p_professional_id, p_expires_hours DEFAULT 2)
- INSERT em `qr_codes` (professional_id, token, url, expires_at)
- `token` = `gen_random_uuid()::TEXT`
- `url` = URL canônica local/prod — usar base configurável ou padrão:
  - `http://localhost:8765/qr/?token={TOKEN}` (dev)
  - Documentar que Leandro troca base depois
- Default 2h expiração
- `GRANT EXECUTE` TO `anon`, `authenticated` (OK dev — restringir Semana 4)

### 1.5 Índice (performance)
```sql
CREATE INDEX IF NOT EXISTS idx_reviews_qr_token ON public.reviews(qr_token) WHERE qr_token IS NOT NULL;
```
